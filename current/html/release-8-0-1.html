E.279. リリース8.0.1

<title>Release 8.0.1</title>

<title>Release date:</title> リリース日: 2005-01-31

This release contains a variety of fixes from 8.0.0, including several security-related issues. For information about new features in the 8.0 major release, see <xref linkend="release-8-0"/>. このリリースは、複数のセキュリティ関連の問題を含む、8.0.0の各種不具合を修正したものです。 8.0メジャーリリースにおける新機能についてはE.280を参照してください。

E.279.1. バージョン8.0.1への移行

<title>Migration to Version 8.0.1</title>

A dump/restore is not required for those running 8.0.0. 8.0.0からの移行ではダンプ/リストアは不要です。

E.279.2. 変更点

<title>Changes</title>
<listitem><para>Disallow <command>LOAD</command> to non-superusers</para>

  • スーパーユーザ以外のLOADを不許可としました

    On platforms that will automatically execute initialization functions of a shared library (this includes at least Windows and ELF-based Unixen), <command>LOAD</command> can be used to make the server execute arbitrary code. Thanks to NGS Software for reporting this.</para></listitem> 共有ライブラリの初期化関数を自動的に行うプラットフォーム(少なくともWindowsやELFベースのUnixを含みます)では、LOADを使用して任意のコードを実行させることができました。 この問題を報告したNGSソフトウェアに感謝します。

    • <listitem><para>Check that creator of an aggregate function has the right to execute the specified transition functions</para>

  • 集約関数の作成者が指定した遷移関数を実行する権利を持つかどうかを検査します

    This oversight made it possible to bypass denial of EXECUTE permission on a function.</para></listitem> この見落としのため、関数のEXECUTE権限を回避することができてしまいました。

    • <listitem><para>Fix security and 64-bit issues in contrib/intagg</para></listitem>

  • contrib/intaggにおけるセキュリティと64ビット問題を修正しました

    • <listitem><para>Add needed STRICT marking to some contrib functions (Kris Jurka)</para></listitem>

  • 一部のcontrib関数で必要であったSTRICT印を追加しました(Kris Jurka)

    • <listitem><para>Avoid buffer overrun when plpgsql cursor declaration has too many parameters (Neil)</para></listitem>

  • plpgsqlのカーソル定義の括弧が多すぎた場合のバッファオーバーランを防止しました (Neil)

    • <listitem><para>Make <command>ALTER TABLE ADD COLUMN</command> enforce domain constraints in all cases</para></listitem>

  • すべての場合においてALTER TABLE ADD COLUMNがドメイン制約を強制するようにしました

    • <listitem><para>Fix planning error for FULL and RIGHT outer joins</para>

  • FULLおよびRIGHT外部結合用の計画作成エラーを修正しました

    The result of the join was mistakenly supposed to be sorted the same as the left input. This could not only deliver mis-sorted output to the user, but in case of nested merge joins could give outright wrong answers. 結合結果が間違って、左側入力と同じようにソートされたものとみなされていました。 これはユーザに間違ったソート結果を導くだけでなく、入れ子状マージ結合の場合は全く間違った回答を与えるかもしれませんでした。

    • <listitem><para>Improve planning of grouped aggregate queries</para></listitem>

  • グループ化された集約関数の計画作成を改良しました

    • <listitem><para><command>ROLLBACK TO <replaceable>savepoint</replaceable></command> closes cursors created since the savepoint</para></listitem>

  • ROLLBACK TO savepointで、セーブポイント以降に作成されたカーソルをクローズします

    • <listitem><para>Fix inadequate backend stack size on Windows</para></listitem>

  • Windowsにおいて不適切なバックエンドスタックサイズを修正しました

    • <listitem><para>Avoid SHGetSpecialFolderPath() on Windows (Magnus)</para></listitem>

  • WindowsにおいてSHGetSpecialFolderPath() を取りやめました(Magnus)

    • <listitem><para>Fix some problems in running pg_autovacuum as a Windows service (Dave Page)</para></listitem>

  • Windowsサービスとして実行する場合のpg_autovacuumの問題を修正しました(Dave Page)

    • <listitem><para>Multiple minor bug fixes in pg_dump/pg_restore</para></listitem>

  • pg_dump/pg_restoreにおける複数の小さな不具合を修正しました

    • <listitem><para>Fix ecpg segfault with named structs used in typedefs (Michael)</para></listitem>

  • typedefsで名前付き構造体が使用された場合のecpgのsegfaultを修正しました(Michael)
